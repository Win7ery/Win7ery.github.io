# 溯源图（Provenance Graph）入门指南

# 什么是溯源图

# 基本定义

溯源图（Provenance Graph），也称为数据来源图或因果图（Causal Graph），是一种有向无环图（DAG）或有向图，用于记录操作系统中实体（entity）之间的信息流和因果关系。它通过捕获系统级事件（如进程创建、文件读写、网络通信等），构建出一张描述"谁对谁做了什么"的全景图。

在网络安全领域，溯源图被广泛用于**高级持续性威胁（APT）**的检测、攻击调查和取证分析。其核心思想是：通过追踪系统实体之间的因果依赖关系，还原攻击链路，识别恶意行为。

# 图的基本元素

溯源图包含两类核心要素：

节点（Node）：代表系统中的实体，通常分为三类：

  • 进程（Process）：操作系统中运行的程序实例，通常由 PID、进程名等属性标识。
  • 文件（File）：磁盘上的文件对象，由路径标识。
  • 网络连接（Socket / NetFlow）：网络通信端点，由 IP 地址和端口标识。

边（Edge）：代表实体之间的系统调用或操作，常见的边类型包括：

边类型 源节点 目标节点 含义
read Process File 进程读取文件
write Process File 进程写入文件
execute Process File 进程执行文件
fork / clone Process Process 进程创建子进程
connect Process Socket 进程发起网络连接
recv / send Socket Process / Process → Socket 网络数据收发

# 溯源图的数据来源：系统审计日志

溯源图的构建依赖于系统审计日志（Audit Log），常见的审计框架包括：

  • Linux Audit（auditd）：Linux 内核级审计系统，记录系统调用。
  • Windows ETW（Event Tracing for Windows）：Windows 的事件追踪机制。
  • CamFlow：基于 Linux Security Module 的信息流追踪框架。

审计日志中的每一条记录通常包含：时间戳、操作主体（subject）、操作客体（object）、操作类型（operation）等字段。将这些记录解析后，即可构建溯源图。

# 为什么需要溯源图

# 传统检测方法的局限

传统的入侵检测系统（IDS）依赖单条日志或单个事件的模式匹配，存在以下问题：

  • 缺乏上下文：单条日志无法反映攻击的完整因果链。
  • 高误报率：孤立事件难以区分正常行为和恶意行为。
  • APT 检测困难：高级攻击通常分阶段执行（侦察→入侵→横向移动→数据窃取），单一规则难以覆盖。

# 溯源图的优势

  • 因果关联：将离散事件串联成因果链，提供攻击行为的全局视图。
  • 攻击还原：支持对攻击路径的完整重建，便于取证分析。
  • 异常检测：通过图结构和语义特征，识别偏离正常行为模式的子图。

# 标准数据集

研究社区提供了多个公开数据集来推动溯源图检测方法的发展。以下是最常用的四个数据集。

# 3.1 DARPA TC（Transparent Computing）计划

DARPA TC 计划是溯源图研究领域最重要的数据来源，由美国国防高级研究计划局（DARPA）资助，旨在开发针对 APT 攻击的检测技术。

# DARPA E3（Engagement 3）

  • 发布时间：2018 年
  • 规模：涵盖数天的操作系统审计日志，包含多个 APT 攻击场景。
  • 操作系统：包括 Linux（Ubuntu）、Windows 和 FreeBSD。
  • 数据格式：基于 CDM（Common Data Model）的 JSON 格式日志，通过 Kafka 发布。
  • 攻击场景：模拟了多种 APT 攻击链，包括鱼叉钓鱼、后门植入、数据窃取等。
  • 特点：
    • 包含正常活动（benign）和攻击活动（malicious）的混合数据。
    • 提供了 ground truth 标注，便于评估。
    • 数据量较大，单个主机的日志可达数百万条事件。

# DARPA E5（Engagement 5）

  • 发布时间：2020 年
  • 改进：相比 E3，E5 引入了更复杂和隐蔽的攻击场景。
  • 操作系统：Linux 和 Windows。
  • 数据格式：同样基于 CDM 模型，但做了若干扩展。
  • 攻击场景：包含更多的横向移动、权限提升和多阶段攻击。
  • 特点：
    • 攻击更加隐蔽，对检测方法的要求更高。
    • 正常活动的背景噪声更大。
    • 被广泛用于评估图神经网络等深度学习方法的有效性。

# 数据获取

DARPA TC 数据集可通过以下渠道获取：

  • GitHub 仓库（darpa-i2o/Transparent-Computing）中有数据说明和部分样本。
  • 完整数据集通常需要通过相关研究机构或项目合作获取。

# OpTC（Operationally Transparent Cyber）

  • 发布机构：DARPA 资助，由 Five Directions 团队收集。
  • 规模：大规模企业网络环境，包含约 1000 台主机、数十亿条事件。
  • 操作系统：主要为 Windows 环境。
  • 数据格式：eCAR（extended Cyber Analytics Repository）格式。
  • 攻击场景：模拟了企业网络中的 APT 攻击，包括红队对蓝队的攻防演练。
  • 特点：
    • 数据规模远超 DARPA E3/E5，更接近真实企业环境。
    • 日志量巨大，对处理效率要求极高。
    • 提供了红队活动的时间线标注。
    • 适合评估方法的可扩展性（scalability）。

# NodLink

  • 来源：由相关学术机构发布的标注数据集。
  • 特点：
    • 提供了节点级别（node-level）和边级别（edge-level）的标注。
    • 标注粒度较细，适合评估细粒度检测方法。

# 溯源图的构建与预处理

# 从日志到图

原始审计日志到溯源图的构建流程如下：

1

原始日志 → 日志解析 → 实体识别 → 关系抽取 → 图构建 → 图压缩/简化

日志解析：将不同格式的审计日志（CDM、eCAR、auditd 等）解析为统一的结构化表示。

实体识别：识别日志中的进程、文件和网络连接实体，建立唯一标识。

关系抽取：将系统调用映射为图中的有向边。

图构建：将实体和关系组装为有向图，通常使用 NetworkX（Python）、Neo4j 或自定义数据结构。

# 图压缩与简化

由于原始溯源图通常非常庞大，需要进行压缩和简化：

  • 节点合并：将同一进程的多个版本（如 fork 前后）合并为同一节点。
  • 边聚合：将同类型的重复边（如多次读取同一文件）合并为一条带权重的边。
  • 噪声过滤：移除与分析无关的系统库加载、临时文件操作等噪声事件。
  • 时间窗口划分：将长时间跨度的图切分为多个时间窗口。

# 常用工具

  • NetworkX：Python 图分析库，适合小规模溯源图。
  • Neo4j：图数据库，适合大规模图的存储和查询。
  • DGL（Deep Graph Library）：适合与图神经网络配合使用。
  • PyG（PyTorch Geometric）：另一个主流的图神经网络框架。

# 检测方法

溯源图上的威胁检测方法可以按照技术路线大致分为以下几类。

# 基于规则和查询的方法

核心思想：定义攻击行为的图模式（graph pattern），在溯源图中进行子图匹配。

代表工作：

  • SLEUTH（USENIX Security 2018）：利用基于标签的信息流追踪，在溯源图中识别攻击入口点并进行前向追踪，构建攻击场景图。
  • Morse（S&P 2019）：引入上下文感知的因果关系分析，通过为信息流标记标签和优先级来缓解依赖爆炸问题。
  • RapSheet（S&P 2020）：将多个威胁告警关联到溯源图上的因果链，构建战术级攻击故事（Tactical Provenance Graph）。

优缺点：

  • 优点：可解释性强，低误报（如果规则设计合理）。
  • 缺点：需要先验知识定义规则，难以检测未知攻击，规则维护成本高。

# 基于统计和异常检测的方法

核心思想：建立正常行为的统计模型，将偏离正常模式的行为标记为异常。

代表工作：

  • StreamSpot（NDSS 2016）：将溯源图分解为固定大小的 shingling 特征向量，使用聚类算法检测异常图。
  • ProvDetector（NDSS 2020）：提取溯源图中的因果路径（causal path），使用自编码器学习正常路径的表示，检测异常路径。
  • Unicorn（NDSS 2020）：使用图的 sketch（近似直方图）来捕获溯源图的演化特征，基于聚类进行异常检测。

优缺点：

  • 优点：无需攻击先验知识，能检测未知攻击。
  • 缺点：误报率较高，异常不一定等于恶意。

# 基于图神经网络（GNN）的方法

核心思想：利用图神经网络学习节点或子图的表示，通过分类或异常检测识别恶意行为。

代表工作：

  • ThreaTrace（USENIX Security 2022）：在溯源图上使用 GNN 进行节点级分类，将每个系统实体标记为良性或恶意。使用 GraphSAGE 作为骨干网络，支持在线检测。
  • MAGIC（USENIX Security 2024）：提出多粒度（节点级 + 图级）的表示学习框架，利用掩码自编码器（Masked Autoencoder）进行自监督预训练，再进行下游检测任务。
  • Kairos（S&P 2024）：将时序溯源图建模为时序图快照序列，使用时序图神经网络捕获图结构的时间演化特征，实现对 APT 攻击的实时检测和重建。
  • Flash（USENIX Security 2024）：引入概念漂移自适应机制，解决溯源图检测中因系统行为变化导致的模型退化问题。

优缺点：

  • 优点：自动学习特征，能捕获复杂的结构和语义模式。
  • 缺点：训练需要大量标注数据（或精心设计的自监督任务），模型可解释性较差，计算开销较大。

# 基于图嵌入和表示学习的方法

核心思想：将溯源图中的节点、边或子图嵌入到低维向量空间，在嵌入空间中进行检测。

代表工作：

  • Shadewatcher（CCS 2022）：将溯源图中的系统调用建模为推荐系统中的交互，使用知识图嵌入方法学习表示。
  • ProvNinja / DISTDET：利用分布式图计算框架处理大规模溯源图上的嵌入学习。

# 基于大语言模型（LLM）的方法（前沿方向）

近年来，部分研究开始探索利用大语言模型辅助溯源图分析：

  • 将溯源图的结构化信息转化为自然语言描述，利用 LLM 的推理能力进行攻击识别。
  • 利用 LLM 生成攻击行为的描述，辅助规则编写和特征设计。
  • 这一方向仍处于早期探索阶段。

# 评估方法与指标

# 基础评估指标

溯源图检测方法的评估通常采用以下标准指标：

  • 精确率（Precision）：P = TP / (TP + FP)，检测结果中正确的比例。
  • 召回率（Recall）：R = TP / (TP + FN)，实际攻击中被正确检测的比例。
  • F1 值：F1 = 2PR / (P + R)，精确率和召回率的调和平均。
  • 误报率（FPR）：FPR = FP / (FP + TN)，正常行为被错误标记为恶意的比例。

# 评估粒度

评估粒度是溯源图检测评估中一个关键但常被忽视的问题。不同方法可能在不同粒度上进行评估：

  • 节点级（Node-level）：评估每个系统实体是否被正确分类为良性/恶意。
  • 边级（Edge-level）：评估每条系统调用边是否被正确识别。
  • 图级（Graph-level）：评估整个溯源图（或子图）是否被正确分类。
  • 场景级（Scenario-level）：评估是否成功识别出完整的攻击场景。

不同粒度下的评估结果可能差异很大，比较不同方法时需注意粒度的一致性。

# Orthrus 评估框架

Orthrus 是近年来提出的一个系统性评估框架，旨在解决溯源图检测领域评估不规范、难以公平比较的问题。

# Orthrus 提出的核心问题

  1. 评估不一致：不同论文在不同数据集、不同粒度、不同划分方式下评估，结果不可比。
  2. 标注质量：部分数据集的 ground truth 标注粒度过粗或存在错误，影响评估可靠性。
  3. 数据泄露：训练集和测试集之间可能存在时序泄露或信息泄露。
  4. 过拟合特定数据集：方法可能在特定数据集上表现良好但泛化能力差。

# Orthrus 的评估维度

Orthrus 从以下维度建立统一的评估框架：

  • 检测有效性（Effectiveness）：在统一的数据划分和指标下比较检测性能。
  • 检测时效性（Timeliness）：衡量从攻击发生到检测触发的延迟。
  • 可扩展性（Scalability）：评估方法在大规模数据上的处理效率。
  • 鲁棒性（Robustness）：评估方法对对抗性攻击（adversarial attack）和概念漂移（concept drift）的抵抗能力。
  • 泛化性（Generalizability）：在不同数据集和攻击场景之间的迁移能力。

# Orthrus 的实践建议

  • 统一数据划分：按时间顺序划分训练集和测试集，避免未来信息泄露。
  • 多数据集评估：在至少两个以上数据集上报告结果。
  • 多粒度报告：同时报告节点级和图级性能。
  • 消融实验：通过消融实验验证各组件的贡献。
  • 运行效率：报告训练时间、推理时间和内存占用。

# 评估中的常见陷阱

数据不平衡：溯源图中恶意节点/边远少于良性节点/边，直接使用准确率（Accuracy）会产生误导。应重点关注 Precision、Recall 和 F1。

时序泄露：使用未来数据训练模型来检测过去的攻击，导致性能虚高。应严格按时间划分数据。

Cherry-picking：只展示表现最好的攻击场景，忽略检测失败的案例。应报告所有场景的平均性能。

标注不一致：不同研究对同一数据集使用不同的标注策略，导致结果不可比。应明确说明标注来源和处理方式。

# 前沿研究方向

# 实时在线检测

大多数方法在离线环境下评估，但实际部署需要支持流式日志的实时处理。如何在保持检测精度的同时实现低延迟检测是关键挑战。

# 跨主机攻击检测

APT 攻击通常涉及多台主机的横向移动。如何将多台主机的溯源图融合分析，检测跨主机攻击链，是重要的开放问题。

# 对抗鲁棒性

攻击者可能有意规避溯源图检测（如混淆系统调用、注入噪声行为）。如何设计对抗鲁棒的检测方法值得关注。

# 概念漂移适应

系统行为会随软件更新、用户习惯变化等因素而改变。如何让检测模型持续适应新的正常行为模式，避免误报激增，是实际部署中的核心难题。

# 可解释性

基于深度学习的方法虽然性能优越，但缺乏可解释性，安全分析师难以理解检测结果。如何生成可解释的检测报告（如标注攻击子图、生成攻击叙述）是活跃的研究方向。

# 图压缩与高效存储

企业环境中每天产生的审计日志可达 TB 级别，如何高效存储和压缩溯源图，同时保留攻击检测所需的信息，仍有很大研究空间。

# 入门论文（建议按顺序阅读）

  1. 综述论文：先阅读一篇溯源图安全分析的综述，建立全局视角。 
    1

    Inam, M. A., Chen, Y., Goyal, A., Liu, J., Mink, J., Michael, N., ... & Hassan, W. U. (2023, May). Sok: History is a vast early warning system: Auditing the provenance of system intrusions. In 2023 IEEE Symposium on Security and Privacy (SP) (pp. 2620-2638). IEEE.
  2. ThreaTrace（2022）和Flash（2024）：基于节点关键属性（节点类型）重构的检测方法。
  3. Kairos（2024）和MAGIC（2024）：基于边关键属性重构的检测方法。
  4. Orthrus（2025）：理解评估框架和方法比较。

# 数据集实践建议

  1. 从 DARPA E3 开始实践，数据规模适中，社区支持丰富。
  2. 使用 Python + NetworkX 解析 CDM 日志，构建第一个溯源图。
  3. 可视化溯源图（推荐 Graphviz 或 Gephi），直观理解图结构。
  4. 尝试复现一个基线方法（如 flash、orthrus等），理解完整的实验流程。
